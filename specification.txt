YouTube 多视角 Archive Sync Controller
需求书 v0.2

1. 项目概述

1.1 项目背景

在观看 VTuber 联动 Archive 时，重度用户通常已经打开多个 YouTube watch tab，并依赖既有生态（账号登录、画质策略、广告过滤插件、快捷键习惯）。

v0.1 的“聚合页直接 embed 多播放器”虽然可实现同屏，但在以下方面存在明显限制：

- 高画质/码率策略不可控，体验不如原生 watch page
- 与用户现有浏览器生态（登录态、插件能力）耦合弱
- 与“用户已开多个 tab 观看”的真实路径不完全一致

因此，v0.2 将产品主路线调整为 Extension-first：

- 主模式：控制用户现有 YouTube tabs（TabSync Mode）
- 可选模式：在扩展独立页中提供 embed 预览网格（Embed Grid Mode）

1.2 产品定义（一句话）

这是一个以 Chrome Extension 为核心的 YouTube Archive 多路同步控制器，优先控制用户已打开的 YouTube tabs，实现统一播放、时间同步与音轨管理；并可选提供 embed 预览模式。


2. 核心决策（相对 v0.1 变更）

2.1 架构路线变更

从“聚合页内多 iframe 播放”为主，改为“Extension 控制现有 watch tabs”为主。

2.2 同屏能力定位变更

- 同屏不再是主模式刚性要求
- 用户可自行排布窗口（系统不接管布局）
- 如需同屏，使用可选 Embed Grid Mode

2.3 范围变更

- 删除“禁止控制现有 tab”的约束（v0.1 的 8.6 不再适用）
- 新增扩展权限、tab 生命周期、content script 注入等需求


3. 产品目标与非目标

3.1 核心目标

系统应让用户能够：

- 从当前浏览器中导入多个 YouTube watch tabs
- 统一控制多路 play/pause/seek/sync
- 对每一路进行 offset/mute/volume/solo 管理
- 通过轻量纠偏减少漂移
- 在不改变用户观看页面与插件生态的前提下使用

3.2 产品定位

- YouTube Archive 多路同步控制工具
- 面向重度用户与半专业回看用户
- 手动配置优先，不依赖自动识别

3.3 非目标

本阶段不追求：

- 自动识别联动关系
- 自动推导最优 offset
- 跨平台（Twitch/Bilibili/Niconico）
- 高级音频处理（mix bus、EQ、compressor）
- 云端账号系统


4. 使用模式定义

4.1 主模式：TabSync Mode（必做）

定义：

- 用户正常打开多个 YouTube watch tabs
- Extension 独立页面仅作为“控制台”
- 多路画面排版由用户自行在浏览器窗口层处理
- 系统通过 content script 控制各 tab 播放行为

优势：

- 最大化复用原生 watch page 能力
- 与用户登录态和插件生态兼容
- 避免 embed 模式在清晰度与行为上的限制

4.2 可选模式：Embed Grid Mode（后续）

定义：

- Extension 独立页中创建多路 iframe 网格作为预览/备选模式
- 保留统一控制逻辑，但明确是次要模式

说明：

- 不承诺与原生 watch page 完全一致体验
- 仅作为“快速对比同屏”的补充能力


5. 系统架构（v0.2）

5.1 Extension 组成

A. Extension Controller Page（独立页面）

- 主 UI（类似 Bitwarden 独立页形态）
- 会话管理与全局状态管理
- 同步引擎运行主体
- 向各 tab 下发控制命令并接收状态回传

B. Background Service Worker

- tab 枚举与生命周期监听（create/update/remove/activate）
- 权限与注入编排
- 消息路由与异常恢复

C. Content Script（每个受控 YouTube tab）

- 读取当前播放状态
- 执行播放控制命令（play/pause/seek/volume/mute/rate）
- 周期上报状态（currentTime、duration、state）

5.2 通信模型

- Controller Page <-> Background: runtime messaging
- Background <-> Content Script: tabs messaging
- 数据流以“路由 routeId”为主键

5.3 时间模型

- 维持 virtual master clock
- 每路目标时间：targetTime = masterTime + offset
- 主参考路用于初始化与用户理解，不作为唯一真值


6. 功能需求（主模式）

6.1 视频源导入与管理

系统应支持：

- 扫描当前窗口或全部窗口中的 YouTube watch tabs
- 将选中 tab 导入当前会话
- 展示每一路基础信息（tab title、video title、url、状态）
- 删除某一路控制（仅移出会话，不关闭 tab）

6.2 全局控制

系统应支持：

- Play All
- Pause All
- Seek All（输入全局 T）
- Sync Now（立即按目标时间重对齐）
- Read Offsets（从已导入 tabs 当前播放时间反向计算并回写 offset）

行为：

- seek 时每路跳转至 T + offset
- 单路失败不阻塞其他路

6.3 Offset 管理

每一路应支持：

- 正/负 offset（小数秒）
- 修改后立即影响目标时间计算
- 支持极端值输入，执行时做边界裁切
- 支持通过 `Read Offsets` 以“当前实际对齐状态”为基准批量回写 offset

6.4 音频控制

每一路必须支持：

- mute（不覆盖 volume 值）
- volume（0~100）
- solo（MVP 单选）
- volume 调整采用 slider，拖动时 UI 实时反映，并以 debounce 下发控制命令

solo 规则：

- 开启某一路 solo 时，其他路强制静音
- 其他路原有 mute/volume 存储状态必须保留
- 取消 solo 后恢复各路原状态效果

6.5 状态显示

每一路应显示：

- 播放状态（loading/ready/playing/paused/buffering/error）
- 当前时间
- offset
- drift
- sync status（synced/minor-drift/soft-correcting/hard-correcting/severe-drift）

6.6 漂移检测与 Auto Sync Correction

系统应周期检查：

- 检查周期建议 500ms
- drift = (masterTime + offset) - currentTime

术语约定：

- 本文中 `Auto Sync Correction` 指“周期性漂移检测 + 自动纠偏”机制

纠偏策略：

- 小误差：稳定区间内不处理（例如 <100ms）
- 中误差：soft correction（playbackRate，建议 0.98~1.02）
- 大误差：hard correction（seek，建议阈值约 1.0s）

约束：

- 避免高频震荡
- 对 hard seek 设置 cooldown

6.7 Auto Focus（可选开关）

系统应支持：

- Controller 中提供 `Auto Focus` 开关（默认关闭）
- 开启后，点击 imported route card 的非交互区域，视为该路被 focus
- focus 后卡片显示明显 border glow
- 同时激活对应 tab，并将其所在 browser window 置前

触发约束：

- 点击 `button/input/select/textarea/a/label/contenteditable` 区域不触发 auto focus
- 关闭 `Auto Focus` 时不触发卡片跳转行为


7. 可选模式需求（Embed Grid）

7.1 能力边界

- 仅作为可选展示模式
- 不作为高画质与插件兼容的主承诺能力

7.2 继承能力

应复用主模式核心逻辑：

- master clock
- offset/sync engine
- audio control 语义
- session 管理


8. 权限与安全

8.1 Chrome 扩展权限（草案）

- tabs
- scripting
- storage
- host_permissions: https://www.youtube.com/*

8.2 安全原则

- 只控制用户明确导入的 tabs
- 不采集与需求无关的隐私数据
- 错误日志默认本地化，不上传云端


9. 异常与边界

9.1 content script 注入失败

- 标记该路不可控
- 不影响其他路
- 允许用户重试注入

9.2 tab 被用户关闭或跳转到非 YouTube 页面

- 自动移出或标记为失效
- 若为主参考路，自动重选或提示用户重选

9.3 广告与播放器状态突变

- 允许出现暂时不可控状态
- 系统应显示该路异常并支持手动 Sync Now 恢复

9.4 跨窗口场景

- 默认可控制多窗口 tabs
- 窗口位置与排版由用户自行管理
- `Auto Focus` 允许跨窗口激活目标 tab 并切换到对应 window


10. 版本规划

10.1 Milestone 1（Extension MVP）

目标：

- 完成 TabSync Mode 的最小闭环

包含：

- Extension Controller Page
- 导入 YouTube tabs
- 全局 play/pause/seek/sync
- 每路 offset/mute/volume/solo
- 基础 sync engine（hard + soft correction）

不包含：

- Embed Grid Mode
- 自动识别联动
- 云端同步

10.2 Milestone 2（可用性增强）

包含：

- session 保存/加载
- 批量导入与筛选
- 更强状态可视化与故障诊断
- 快捷操作（如主路有声）

10.3 Milestone 3（可选模式与高级增强）

包含：

- Embed Grid Mode
- 配置导入导出
- 分享会话配置


11. MVP 验收标准（TabSync Mode）

满足以下条件可视为 MVP 达标：

- 能从浏览器导入多个 YouTube tabs
- 多路可统一 play/pause/seek/sync
- 每路 offset 生效且可观察到同步结果
- `Read Offsets` 可从 tabs 当前状态反向回写 offset
- mute/volume/solo 语义正确
- 小误差软纠偏、大误差硬拉回可工作
- 开启 `Auto Focus` 后可通过卡片点击快速切换到目标 tab/window
- 单路异常不影响其他路


12. 术语

- Route：会话中的一路受控视频
- Master Clock：系统内部统一时间基准
- Main Reference Route：主参考路，仅用于初始化与认知锚点
- TabSync Mode：控制现有 YouTube tabs 的主模式
- Embed Grid Mode：扩展页内 iframe 网格的可选模式
- Sync Now：手动触发的一次性立即重对齐
- Auto Sync Correction：后台周期运行的持续纠偏机制
- Read Offsets：从已导入 routes 的当前时间反向计算 offset 并批量回写
- Auto Focus：点击 route card 时自动聚焦对应 tab/window 的快速切换机制
